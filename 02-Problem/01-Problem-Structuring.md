# Problem Structuring

## 1. Purpose
This file defines how a raw venture problem is decomposed into structured components within the Problem layer.

It ensures problems are not treated as single statements, but as analyzable problem spaces.

---

## 2. Core Principle
A problem is not atomic.
It must be broken into structured sub-components to enable validation and reasoning.

---

## 3. Structuring Dimensions
Every problem MUST be decomposed along the following axes:

### 3.1 User Dimension
- Who experiences the problem
- Which segment or role is affected

### 3.2 Context Dimension
- In what situation the problem occurs
- Environmental or operational conditions

### 3.3 Pain Dimension
- What failure, friction, or unmet need exists
- Severity and frequency of impact

### 3.4 Constraint Dimension
- What limits current solutions from resolving the problem
- Economic, behavioral, systemic constraints

---

## 4. Decomposition Rule
A valid structured problem must:
- Be split into at least one dimension
- Avoid solution embedding
- Remain independent of implementation

---

## 5. Output Format Principle
Problem structuring produces:
- Structured problem representation
- Not solutions
- Not product definitions

---

## 6. Role in OS
This layer transforms a single problem statement into a multi-dimensional problem space suitable for validation and downstream reasoning.
