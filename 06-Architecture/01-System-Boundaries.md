# System Boundaries

## 1. Purpose
Defines the conceptual boundary of the system being modeled within the venture operating system.

It clarifies what is included inside the system scope and what remains external.

---

## 2. Boundary Principle
A system is defined not by its internal components, but by its boundary of responsibility.

---

## 3. Internal Scope
The system includes:
- actors directly participating in value creation
- entities managed or influenced by the system
- processes executed within system control

---

## 4. External Scope
The system excludes:
- market dynamics not directly influenced by the system
- external platforms and infrastructures
- competitors and independent actors

---

## 5. Interfaces
System interacts with external environment through defined interfaces:
- users
- partners
- external services

---

## 6. Inclusion Criteria
An element is inside the boundary if:
- it is directly required for value creation
- or it is directly controlled or governed by the system

---

## 7. Exclusion Criteria
An element is outside the boundary if:
- it has no direct operational dependency
- it only influences indirectly at macro level

---

## 8. Dependency
Derived from:
- Problem layer
- Validation layer
- Product hypothesis
- Domain model

---

## 9. Role in Architecture
System boundaries define the input frame for architectural decomposition and component modeling.
